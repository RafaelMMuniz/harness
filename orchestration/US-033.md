# US-033: Add saved analyses feature

**Priority:** 33
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, analysts can save and reload trend or funnel configurations by name, avoiding the need to re-enter parameters each time. This is a Tier 4 nice-to-have (BR-400).

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- New database table: saved_analyses with columns: id (INTEGER PRIMARY KEY AUTOINCREMENT), name (TEXT NOT NULL), type ('trend' | 'funnel'), config (TEXT, JSON string storing all query parameters), created_at (TEXT), updated_at (TEXT)
- POST /api/saved-analyses accepts { name: string, type: 'trend' | 'funnel', config: object } and stores it, returns the saved record with id
- GET /api/saved-analyses returns all saved analyses sorted by updated_at descending
- GET /api/saved-analyses/:id returns a single saved analysis
- DELETE /api/saved-analyses/:id deletes a saved analysis
- On the Trends page: a 'Save' button in the control bar that opens a dialog asking for a name, saves the current query config (event name, measure, property, breakdown, date range, granularity, chart type)
- On the Funnels page: a 'Save' button that saves the current funnel config (steps, date range)
- A 'Saved Analyses' dropdown or sidebar section on both pages that lists saved analyses of the matching type, clicking one loads its config into the controls and re-runs the query
- Uses shadcn/ui Dialog, Input, and Button components
- Typecheck passes
- Verify in browser using dev-browser skill
- Saved analyses API tests pass: npx playwright test api-saved.spec.ts
