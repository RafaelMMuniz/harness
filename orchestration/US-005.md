# US-005: Implement identity resolution query logic

**Priority:** 5
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, all queries that filter or count by user identity go through a resolution layer that merges anonymous device events with their identified user. This is the most critical story in the entire system — read BR-101 in CLAUDE.md for the full specification.

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- A resolveIdentity(deviceOrUserId: string) function exists that: given a user_id, returns that user_id; given a device_id, looks up the identity_mappings table and returns the mapped user_id if one exists, otherwise returns the device_id as-is
- A getEventsForUser(userId: string) function exists that returns all events attributed to a resolved user: events where user_id matches, PLUS events where device_id is mapped to that user_id via identity_mappings
- GET /api/events?user_id=<id> returns all resolved events for that user, sorted by timestamp ascending (oldest first — this is the user timeline view, distinct from the paginated list in US-007 which sorts descending/newest first)
- GET /api/events?device_id=<id> resolves the device to a user (if mapping exists) and returns all events for that resolved user, sorted by timestamp ascending
- Response includes an array of event objects with parsed properties (JSON object, not string)
- The merge is retroactive: events recorded before the identity mapping was created are included
- Typecheck passes
- All identity resolution tests pass: npx playwright test identity.spec.ts

## Spec anchors (READ these sections if the briefing is unclear)

- prd.json notes — "The SQL query for getEventsForUser needs to join events with identity_mappings: SELECT * FROM events WHERE user_id = ? OR device_id IN (SELECT device_id FROM identity_mappings WHERE user_id = ?)"

## Test scope (for the validator)

- Primary test file: `e2e/US-005.spec.ts` or run existing `e2e/identity.spec.ts`
- Coverage the test MUST have (derived from acceptance criteria above):
  - Retroactive merge: anonymous events before mapping are included after mapping created
  - Multi-device merge: events from multiple devices mapped to same user all appear
  - Device resolution: GET with device_id resolves to user and returns all user events
  - Properties returned as parsed objects, not JSON strings
  - Events sorted by timestamp ascending (oldest first)
- Edge cases worth adversarial testing:
  - Device with no mapping — returns only that device's events, device_id as identity
  - User with no events — returns empty array
  - Collision rejection (409) — already covered by US-003 but verify resolution doesn't return conflicted events

## Risk notes

This is the most critical story. Every downstream query (US-007, US-011, US-015, US-025) depends on identity resolution being correct. The SQL pattern is: `WHERE user_id = ? OR device_id IN (SELECT device_id FROM identity_mappings WHERE user_id = ?)`. Direct `WHERE user_id = ?` without the identity_mappings join is a bug.
