# US-022: Add property metadata API for numeric detection

**Priority:** 22
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, the frontend can query which properties exist for a given event type and whether they are numeric, string, or boolean. This powers the measure/breakdown selectors on the Trends page.

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- GET /api/events/:eventName/properties returns an array of property descriptors for that event type
- Each descriptor has: { name: string, type: 'string' | 'number' | 'boolean', sample_values: string[] }
- Type detection logic: scan a sample of recent events (up to 500) for the given event name, for each property key check if all non-null values parse as numbers — if yes, type is 'number'; if all are 'true'/'false', type is 'boolean'; otherwise 'string'
- sample_values contains up to 10 distinct values for that property (useful for breakdown selectors)
- Properties with no values across all sampled events are excluded
- Response is fast (< 500ms) on the sample dataset
- Typecheck passes
- Property metadata API tests pass: npx playwright test api-properties.spec.ts

## Test scope (for the validator)

- Primary test file: `e2e/US-022.spec.ts` or run `e2e/api-properties.spec.ts`
- Coverage the test MUST have (derived from acceptance criteria above):
  - Returns array of { name, type, sample_values }
  - Correctly detects numeric properties (amount, quantity) vs string properties (url, button_name)
  - Returns 404 or empty array for unknown event name
  - sample_values has up to 10 distinct values
- Edge cases worth adversarial testing:
  - Property with mixed types (some numeric, some string) classified as string
  - Boolean detection (all values 'true'/'false')
  - Property that is null in all events is excluded
