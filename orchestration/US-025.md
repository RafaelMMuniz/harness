# US-025: Add funnel analysis API endpoint

**Priority:** 25
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, growth managers can compute funnel conversion rates for a sequence of 2-5 events, using resolved identities. The API enforces temporal step ordering and handles cross-device identity merges.

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- POST /api/funnels/query accepts a JSON body: { steps: string[] (2-5 event names), start_date: string (ISO 8601), end_date: string (ISO 8601) }
- Validated with Zod: steps must have 2-5 entries, each non-empty string; start_date required; end_date required and must be after start_date
- The funnel uses resolved identities: for each event, resolve the identity via identity_mappings (same logic as trend unique_users)
- A user 'converts' through step N if they performed the step N event AFTER performing the step N-1 event (by timestamp), within the date range
- Response body: { steps: Array<{ event_name: string, count: number, conversion_rate: number, drop_off: number }>, overall_conversion_rate: number }
- steps[0].count is the number of distinct resolved users who performed the first event in the date range
- steps[0].conversion_rate is always 1.0 (100%)
- steps[N].conversion_rate is steps[N].count / steps[0].count (relative to funnel entry, not previous step)
- steps[N].drop_off is steps[N-1].count - steps[N].count
- overall_conversion_rate is steps[last].count / steps[0].count
- A user who performs step 1 anonymously and step 2 after identity merge is counted as ONE user progressing (not a dropout)
- Typecheck passes
- Funnel API tests pass: npx playwright test api-funnels.spec.ts

## Spec anchors (READ these sections if the briefing is unclear)

- prd.json notes — "Algorithm: 1) For each step, get all (resolved_user, timestamp) pairs. 2) Starting from step 1 users, find users who appear in next step with a later timestamp. 3) Use Sets for efficient intersection. Identity resolution JOIN same as trends."

## Test scope (for the validator)

- Primary test file: `e2e/US-025.spec.ts` or run `e2e/api-funnels.spec.ts`
- Coverage the test MUST have (derived from acceptance criteria above):
  - Valid 3-step funnel returns correct counts, conversion_rates, drop_offs
  - steps[0].conversion_rate is always 1.0
  - Conversion counts decrease or stay equal across steps
  - Step order enforced: user doing step 2 before step 1 NOT counted
  - Identity resolution: anonymous step 1 + identified step 2 = one user progressing
  - Fewer than 2 steps returns 400
  - More than 5 steps returns 400
- Edge cases worth adversarial testing:
  - User does step 1 and step 3 but not step 2 — not counted in step 3
  - All users drop off at step 2 — step 2+ counts are 0
  - Empty funnel (no users did step 1) — all counts 0, overall_conversion_rate 0

## Risk notes

This is the most algorithmically complex endpoint. Two critical invariants: (1) temporal ordering — step N must happen AFTER step N-1 by timestamp, and (2) identity resolution — a user doing step 1 as device-X and step 2 as user-Y (after merge) must count as one user progressing. Getting either wrong produces incorrect funnels.
