# US-020: Write automated tests for trend aggregation

**Priority:** 20
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, the codebase has server-side tests verifying that trend time bucketing, zero-filling, and identity-resolved unique counts are correct. This protects the core aggregation logic against regressions.

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- Test file at server/src/__tests__/trends.test.ts
- Test 1 - Daily bucketing: insert events across 3 different days for the same event type, call GET /api/trends with daily granularity, assert each day has the correct total_count and that all days in the range are present (including zero-count days)
- Test 2 - Weekly bucketing: insert events across 2 different weeks, call GET /api/trends with weekly granularity, assert events are grouped into the correct week buckets
- Test 3 - Unique users with identity resolution: insert 5 events from device-A (anonymous), create identity mapping linking device-A to user-B, insert 3 more events from user-B directly, call GET /api/trends, assert unique_users counts user-B as ONE user not two, while total_count is 8
- Test 4 - Zero-fill gaps: insert events only on day 1 and day 5 of a 7-day range, assert response contains 7 entries with days 2-4 and 6-7 having total_count: 0
- Test 5 - Date range filtering: insert events outside the requested range, assert they are excluded from results
- All tests pass
- Typecheck passes
