# US-015: Add trend aggregation API endpoint

**Priority:** 15
**Status:** passes:false (briefing generated during --fresh)

## What this story accomplishes

After this story lands, the frontend can fetch time-bucketed event counts (daily or weekly) with zero-filled gaps and identity-resolved unique user counts. This powers the trend charts.

## Acceptance Criteria (VERBATIM from prd.json — DO NOT paraphrase)

- GET /api/trends returns time-bucketed event counts
- Required query parameter: event_name (string, the event type to aggregate)
- Optional query parameters: start_date (ISO 8601, default 30 days ago), end_date (ISO 8601, default now), granularity ('day' | 'week', default 'day')
- Request is validated with Zod: event_name is required and non-empty, granularity must be 'day' or 'week', start_date must be before end_date if both are provided
- Returns 400 with descriptive error if validation fails
- Response body: { event_name: string, granularity: string, start_date: string, end_date: string, data: Array<{ date: string, total_count: number, unique_users: number }> }
- Each entry in data represents one time bucket (one day or one week starting Monday)
- total_count is the number of events in that bucket
- unique_users is the count of distinct resolved identities in that bucket — anonymous devices that are mapped to a user_id via identity_mappings are counted under that user_id, not separately
- Time buckets with zero events are included in the response with total_count: 0 and unique_users: 0 (no gaps in the series)
- Typecheck passes
- Trend API basic tests pass: npx playwright test api-trends.spec.ts (daily/weekly bucketing, zero-fill, identity resolution tests)

## Spec anchors (READ these sections if the briefing is unclear)

- prd.json notes — "unique_users calculation: use COALESCE(im.user_id, e.user_id, e.device_id) as resolved ID with LEFT JOIN on identity_mappings. COUNT(DISTINCT resolved_id). For weekly buckets use strftime for ISO week start (Monday). Zero-fill in application code."

## Risk notes

The unique_users calculation is the hard part. The SQL needs: `COALESCE(im.user_id, e.user_id, e.device_id)` with `LEFT JOIN identity_mappings im ON e.device_id = im.device_id`. A simple `COUNT(DISTINCT user_id)` is wrong — it misses anonymous devices and double-counts mapped ones.
